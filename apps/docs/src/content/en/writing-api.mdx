# Writing API Documentation

Create comprehensive API documentation that developers love. Learn to use Properties for parameters, Tags for HTTP methods, Libraries for showcasing SDKs, and specialized components that make API references clear and actionable. {{ className: 'lead' }}

## Properties Lists

Document API parameters with `<Properties>` and `<Property>`:

<Properties>
  <Property name="id" type="string">
    Unique identifier for the resource. Automatically generated when the resource is created.
  </Property>

<Property name="name" type="string">
  Display name of the resource. Must be between 1 and 100 characters.
</Property>

<Property name="email" type="string">
  Valid email address. Must be unique across all users.
</Property>

  <Property name="created_at" type="timestamp">
    ISO 8601 timestamp indicating when the resource was created.
  </Property>
</Properties>

```mdx
<Properties>
  <Property name="id" type="string">
    Description of the parameter and its usage.
  </Property>
</Properties>
```

## HTTP Method Tags

Tags automatically get colored based on HTTP methods:

<div className="flex gap-2 flex-wrap">
  <Tag>GET</Tag>
  <Tag>POST</Tag>
  <Tag>PUT</Tag>
  <Tag>DELETE</Tag>
  <Tag color="zinc">CUSTOM</Tag>
  <Tag color="emerald">SUCCESS</Tag>
  <Tag color="rose">ERROR</Tag>
</div>

```mdx
<Tag>GET</Tag>
<Tag>POST</Tag>
<Tag color="emerald">SUCCESS</Tag>
```

## Libraries Component

Showcase official SDKs and libraries using the `<Libraries>` component:

<Libraries />

The Libraries component displays a predefined grid of official libraries. The MDX syntax is simple:

```mdx
<Libraries />
```

**Note**: The Libraries component displays a fixed set of programming languages (PHP, Ruby, Node.js, Python, Go). The actual libraries and links are configured in the component file.

### Libraries Use Cases

- SDK documentation pages
- Getting started guides
- Language-specific integration guides
- Official library showcases

## API Best Practices

- Always document all parameters with Properties components
- Include example requests and responses
- Use proper HTTP status codes with Tags
- Provide clear error messages
- Include authentication requirements
- Use Libraries component for SDK pages
- Keep Properties lists focused and well-organized
